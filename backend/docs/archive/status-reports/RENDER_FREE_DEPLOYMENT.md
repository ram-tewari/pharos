## Step 4: Keep-Alive Setup (Optional but Recommended)

Render Free tier spins down after 15 minutes of inactivity, causing 50-second cold starts. For active testing, set up a keep-alive monitor.

### 4.1 Sign Up for UptimeRobot

1. Go to https://uptimerobot.com
2. Sign up for free account (50 monitors, 5-min intervals)

### 4.2 Create Monitor

1. Click **"Add New Monitor"**
2. Configure:
   - **Monitor Type**: HTTP(s)
   - **Friendly Name**: Pharos Keep-Alive
   - **URL**: `https://pharos-cloud-api.onrender.com/health`
   - **Monitoring Interval**: 5 minutes (free tier)
   - **Alert Contacts**: Your email (optional)

3. Click **"Create Monitor"**

### 4.3 Verify Keep-Alive

UptimeRobot will ping your `/health` endpoint every 5 minutes, keeping the service warm.

**Note**: This is for testing only. In production, use Render's paid tier or implement proper scaling.

## Step 5: Configure Ronin Integration

### 5.1 Get Pharos URL and Token

From Render dashboard:
- **URL**: `https://pharos-cloud-api.onrender.com`
- **Token**: Copy `PHAROS_ADMIN_TOKEN` from environment variables

### 5.2 Configure Ronin

In your Ronin configuration (local RTX 4070 edge worker):

```bash
# .env or config file
PHAROS_API_URL=https://pharos-cloud-api.onrender.com
PHAROS_API_KEY=<YOUR_PHAROS_ADMIN_TOKEN>
PHAROS_TIMEOUT=5000  # 5 seconds (account for cold start)
```

### 5.3 Test Ronin → Pharos Connection

```python
# test_ronin_pharos.py
import requests

PHAROS_URL = "https://pharos-cloud-api.onrender.com"
PHAROS_TOKEN = "your-token-here"

# Test health
response = requests.get(f"{PHAROS_URL}/health")
print(f"Health: {response.json()}")

# Test context retrieval
response = requests.post(
    f"{PHAROS_URL}/api/context/retrieve",
    headers={"X-API-Key": PHAROS_TOKEN},
    json={
        "query": "How does authentication work?",
        "codebase": "myapp-backend",
        "max_chunks": 10
    }
)
print(f"Context: {response.json()}")
```

## Performance Expectations

### Cold Start (First Request After 15 Min)
- **Time**: 50 seconds
- **Reason**: Render spins up container
- **Mitigation**: UptimeRobot keep-alive

### Warm Requests
- **Health Check**: <100ms
- **Context Retrieval**: 800ms - 2s
- **GitHub Ingestion**: 30-60s per repo

### Resource Limits (Free Tier)
- **RAM**: 512MB (sufficient for metadata + embeddings)
- **CPU**: Shared (adequate for API requests)
- **Storage**: NeonDB 512MB (enough for 50-100 repos)
- **Redis**: 256MB (enough for query cache)

## Troubleshooting

### Issue: 502 Bad Gateway

**Cause**: Service is spinning up (cold start)  
**Solution**: Wait 50 seconds and retry

### Issue: Database Connection Error

**Cause**: NeonDB connection string incorrect  
**Solution**: Verify `DATABASE_URL` in Render environment variables

### Issue: Redis Connection Error

**Cause**: Upstash credentials incorrect  
**Solution**: Verify `UPSTASH_REDIS_REST_URL` and `UPSTASH_REDIS_REST_TOKEN`

### Issue: 401 Unauthorized

**Cause**: Missing or incorrect API key  
**Solution**: Include `X-API-Key` header with `PHAROS_ADMIN_TOKEN`

### Issue: Slow Context Retrieval (>5s)

**Cause**: Cold start or large codebase  
**Solution**: 
- Enable keep-alive monitoring
- Reduce `max_chunks` parameter
- Use hybrid GitHub storage (Phase 5)

### Issue: Out of Memory

**Cause**: Too many embeddings in RAM  
**Solution**:
- Reduce `MAX_QUEUE_SIZE` to 5
- Disable `CHUNK_ON_RESOURCE_CREATE`
- Use hybrid GitHub storage (Phase 5)

### When to Upgrade

Upgrade to paid tier when:
- **Cold starts** become problematic (>10 requests/hour)
- **Storage** exceeds 512MB (>100 repos)
- **Redis** hits 10K commands/day (>400 requests/hour)
- **RAM** causes OOM errors (>5 concurrent requests)

## Security Notes

### API Key Protection
- **Never commit** `PHAROS_ADMIN_TOKEN` to Git
- **Rotate keys** regularly (monthly)
- **Use environment variables** only

### Database Security
- NeonDB uses SSL by default (`sslmode=require`)
- Connection pooling enabled
- No public access (API only)

### Redis Security
- Upstash uses TLS by default
- REST API with token authentication
- No direct Redis protocol access
